## 2. 开发工具和相关技术

### 2.1 技术栈概述

本项目采用前后端分离的架构设计，后端基于Java Spring Boot框架，前端使用Vue.js构建Web应用，同时使用uni-app开发跨平台移动应用。系统集成了多种开源技术和工具，形成了一套完整的试验任务管理解决方案。

**核心技术栈如下：**

| 类别    | 技术/框架          | 版本      | 用途                   |
| ----- | -------------- | ------- | -------------------- |
| 后端框架  | Spring Boot    | 2.5.15  | 提供RESTful API和业务逻辑实现 |
| 持久层   | MyBatis        | -       | 数据库操作和ORM映射          |
| 数据库   | MySQL          | -       | 关系型数据存储              |
| 缓存    | Redis          | -       | 缓存和会话管理              |
| 前端框架  | Vue.js         | 2.6.12  | Web应用界面构建            |
| 移动端框架 | uni-app + Vue3 | -       | 跨平台移动应用开发            |
| UI组件库 | Element UI     | 2.15.13 | Web端UI组件             |
| 认证    | JWT            | 0.9.1   | 无状态用户认证              |
| 二维码处理 | Google ZXing   | 3.5.1   | 二维码生成和解析             |
| 报表生成  | JasperReports  | -       | PDF格式报表生成            |
| 模板引擎  | Thymeleaf      | -       | HTML模板渲染             |

### 2.2 系统架构

本项目采用分层架构设计，将系统划分为表示层、业务逻辑层、数据访问层和数据存储层四个主要层次，实现了关注点分离和代码复用。

**系统架构图：**

1. **表示层**：包括Web前端和移动应用端
   - Web前端：基于Vue.js 2.6.12和Element UI 2.15.13构建
   - 移动应用：基于uni-app和Vue3开发，支持多平台运行
2. **业务逻辑层**：基于Spring Boot 2.5.15实现
   - 控制器层：处理HTTP请求，调用业务服务
   - 服务层：实现核心业务逻辑
   - 工具类：提供通用功能支持
3. **数据访问层**：基于MyBatis实现
   - Mapper接口：定义数据库操作方法
   - XML映射文件：配置SQL语句
   - 拦截器：实现乐观锁等横切关注点
4. **数据存储层**：
   - MySQL：存储业务数据
   - Redis：提供缓存支持

### 2.3 后端技术

#### 2.3.1 Spring Boot

Spring Boot 2.5.15作为项目的后端框架，提供了以下核心功能：

- **自动配置**：简化应用配置，减少样板代码
- **内嵌容器**：内置Tomcat服务器，简化部署
- **依赖管理**：通过starter实现依赖的自动管理
- **Actuator**：提供应用监控和管理功能

项目使用Spring Boot实现了RESTful API接口，支持跨域请求处理，并集成了Spring Security进行权限控制。

#### 2.3.2 MyBatis

MyBatis作为持久层框架，负责数据库操作和ORM映射，主要特点包括：

- **SQL映射**：通过XML文件或注解配置SQL语句
- **动态SQL**：支持条件查询和复杂SQL构建
- **缓存机制**：提供一级缓存和二级缓存
- **插件系统**：支持自定义拦截器

项目中使用MyBatis实现了乐观锁并发控制，通过自定义拦截器自动处理版本号字段的更新，确保数据一致性。

#### 2.3.3 JWT认证

使用JWT 0.9.1实现无状态用户认证，主要流程包括：

1. 用户登录时生成JWT令牌
2. 令牌包含用户信息和权限声明
3. 后续请求通过Authorization头携带令牌
4. 服务端验证令牌有效性并提取用户信息

JWT认证机制简化了会话管理，提高了系统的可扩展性。

#### 2.3.4 定时任务

基于Spring Quartz实现定时任务，主要用于：

- **超期预警**：定期检查试验任务和项目的截止时间，发送预警通知
- **数据同步**：定期同步相关系统数据

定时任务配置灵活，支持cron表达式定义执行时间。

### 2.4 前端和移动端技术

#### 2.4.1 Vue.js

Vue.js 2.6.12作为前端框架，提供了以下核心功能：

- **响应式数据绑定**：实现数据与视图的自动同步
- **组件化开发**：提高代码复用性和可维护性
- **虚拟DOM**：提升渲染性能
- **路由管理**：实现单页应用导航

项目中使用Vue.js构建了试验任务管理、超期预警、二维码扫描等功能模块。

#### 2.4.2 Element UI

Element UI 2.15.13作为UI组件库，提供了丰富的界面组件：

- **表单组件**：输入框、选择器、日期选择等
- **表格组件**：数据展示、排序、筛选等
- **对话框组件**：信息提示、确认操作等
- **导航组件**：菜单、面包屑等

Element UI的使用大大简化了前端开发，提高了界面的一致性和美观度。

#### 2.4.3 uni-app

uni-app作为跨平台移动应用框架，基于Vue3实现，支持：

- **多平台发布**：同时支持iOS、Android、H5等平台
- **原生体验**：接近原生应用的性能和体验
- **组件化开发**：与Vue.js开发方式一致
- **二维码扫描**：集成原生扫码功能

项目使用uni-app开发了移动应用，实现了试验任务的移动化管理和二维码扫描功能。

### 2.5 数据存储和文件服务

#### 2.5.1 MySQL

MySQL作为关系型数据库，用于存储项目的核心业务数据：

- **试验任务数据**：任务基本信息、进度、状态等
- **项目数据**：项目基本信息、负责人、截止时间等
- **用户数据**：用户信息、角色、权限等
- **配置数据**：系统参数、字典数据等

数据库设计遵循规范化原则，确保数据的一致性和完整性。

#### 2.5.2 Redis

Redis作为缓存服务，主要用于：

- **会话管理**：存储用户会话信息
- **热点数据缓存**：缓存频繁访问的数据
- **分布式锁**：实现分布式环境下的并发控制

Redis的使用提高了系统的响应速度和并发处理能力。

### 2.6 报表打印技术

项目采用双模式打印策略，支持不同场景的打印需求：

#### 2.6.1 JasperReports

JasperReports用于生成PDF格式的正式报表，主要特点：

- **模板设计**：通过iReport Designer可视化设计报表模板
- **数据填充**：支持多种数据源，包括JDBC、JavaBean等
- **格式控制**：精确控制报表布局和样式
- **导出格式**：支持PDF、Excel、HTML等多种格式

项目中使用JasperReports生成试验任务的正式PDF报表。

#### 2.6.2 Thymeleaf

Thymeleaf作为HTML模板引擎，用于生成Web端打印预览，主要特点：

- **自然模板**：模板文件是有效的HTML文件
- **表达式语言**：支持丰富的表达式语法
- **Spring集成**：与Spring框架无缝集成
- **实时预览**：支持Web端实时预览打印效果

项目中使用Thymeleaf实现了试验任务的Web端打印预览功能。

### 2.7 其他相关技术

#### 2.7.1 二维码技术

使用Google ZXing 3.5.1实现二维码的生成和解析：

- **二维码生成**：将试验任务信息编码为二维码
- **二维码解析**：通过扫码获取任务信息，实现快速定位
- **Base64编码**：将二维码转换为Base64格式，便于在前端展示

二维码技术的应用提高了试验任务管理的便捷性和效率。

#### 2.7.2 乐观锁并发控制

通过MyBatis拦截器实现乐观锁并发控制：

- **版本号机制**：为数据记录添加版本号字段
- **自动版本控制**：拦截器自动处理版本号的检查和更新
- **冲突检测**：当检测到并发冲突时抛出异常

乐观锁机制确保了多用户同时操作时的数据一致性。

#### 2.7.3 RBAC权限控制

基于角色的访问控制（RBAC）：

- **角色定义**：根据功能需求定义不同角色
- **权限分配**：为角色分配相应的操作权限
- **用户-角色关联**：将用户与角色关联，实现权限控制

RBAC机制确保了系统的安全性和操作的合规性。

1. **项目立项阶段**：
   - 项目申请和审批
   - 项目基本信息录入
   - 项目团队组建和职责分配
   - 项目计划制定和时间节点确定
2. **任务分解阶段**：
   - 项目分解为多个试验任务
   - 任务基本信息录入
   - 任务负责人分配
   - 任务时间计划制定
3. **任务执行阶段**：
   - 任务状态更新和进度跟踪
   - 质量信息采集和记录
   - 问题发现和处理
   - 任务变更和调整
4. **任务完成阶段**：
   - 任务结果验证和确认
   - 试验数据整理和归档

#### 3.1.2 质量信息流转卡业务模型

   - 流转路径和节点设置

### 3.5 数据库设计

#### 3.5.1 数据库表结构

**1. 用户表（sys\_user）**

| 字段名          | 数据类型         | 约束          | 描述   |
| ------------ | ------------ | ----------- | ---- |
| user\_id     | bigint       | PRIMARY KEY | 用户ID |
| dept\_id     | bigint       | FOREIGN KEY | 部门ID |
| user\_name   | varchar(30)  | NOT NULL    | 用户名  |
| nick\_name   | varchar(30)  | NOT NULL    | 昵称   |
| password     | varchar(100) | NOT NULL    | 密码   |
| status       | char(1)      | NOT NULL    | 状态   |
| create\_time | datetime     | NOT NULL    | 创建时间 |
| update\_time | datetime     | NOT NULL    | 更新时间 |

**2. 角色表（sys\_role）**

| 字段名          | 数据类型         | 约束          | 描述   |
| ------------ | ------------ | ----------- | ---- |
| role\_id     | bigint       | PRIMARY KEY | 角色ID |
| role\_name   | varchar(30)  | NOT NULL    | 角色名称 |
| role\_key    | varchar(100) | NOT NULL    | 角色键值 |
| status       | char(1)      | NOT NULL    | 状态   |

### 4.4 关键机制设计

#### 4.4.1 基于 MyBatis 拦截器的乐观锁并发控制设计

**设计思路**：

- **版本号机制**：在需要并发控制的表中添加version字段
- **MyBatis拦截器**：通过拦截器自动处理版本号的检查和更新
- **冲突检测**：当检测到并发冲突时抛出异常

**实现原理**：

1. **拦截器配置**：配置MyBatis拦截器，拦截UPDATE和DELETE操作
2. **SQL解析**：解析SQL语句，提取表名和where条件
3. **版本号检查**：自动在where条件中添加版本号检查
4. **版本号更新**：在UPDATE语句中自动更新版本号
5. **冲突处理**：当更新影响行数为0时，抛出乐观锁异常

**代码实现**：

- 创建`OptimisticLockInterceptor`类，实现`Interceptor`接口
- 重写`intercept`方法，处理SQL语句
- 配置拦截器，使其生效

**使用方式**：

- 在需要并发控制的实体类中添加version字段
- 在Mapper接口中定义更新方法
- 拦截器会自动处理版本号的检查和更新

拦截器在 `StatementHandler#prepare` 阶段拦截 UPDATE，若参数对象含 `version` 字段则拼接 `AND version = ?` 条件并在内存中递增版本号，核心逻辑如下（摘自 `ruoyi-framework` 模块 `OptimisticLockInterceptor.java`）：

```java
@Override
public Object intercept(Invocation invocation) throws Throwable {
    StatementHandler statementHandler = (StatementHandler) invocation.getTarget();
    MetaObject metaObject = SystemMetaObject.forObject(statementHandler);
    MappedStatement mappedStatement = (MappedStatement) metaObject.getValue("delegate.mappedStatement");
    SqlCommandType sqlCommandType = mappedStatement.getSqlCommandType();
    if (SqlCommandType.UPDATE.equals(sqlCommandType)) {
        BoundSql boundSql = statementHandler.getBoundSql();
        String originalSql = boundSql.getSql();
        Object parameterObject = boundSql.getParameterObject();
        if (parameterObject != null && hasVersionField(parameterObject)) {
            Integer version = getVersion(parameterObject);
            if (version != null) {
                String newSql = addOptimisticLockCondition(originalSql, version);
                setFieldValue(boundSql, "sql", newSql);
                incrementVersion(parameterObject);
            }
        }
    }
    return invocation.proceed();
}
```

在 `MyBatisConfig` 中将该拦截器注册为 MyBatis 插件后，所有带 `version` 实体的更新语句将自动带上乐观锁条件（摘自 `MyBatisConfig.java`）：

```java
sessionFactory.setPlugins(new OptimisticLockInterceptor());
```

- **日志分析**：分析用户操作行为，发现异常操作

**代码实现**：
- `LogController.java`：处理日志相关的 HTTP 请求
- `LogService.java`：实现日志管理的业务逻辑
- `LogAspect.java`：使用 AOP 实现操作日志的自动记录

**核心方法**：
- `recordLog`：记录操作日志
- `getLogList`：查询日志列表
- `analyzeLog`：分析日志数据

**实现流程**：
1. 用户执行操作，如创建任务、更新项目等
2. AOP 切面拦截操作，记录操作日志
3. 日志数据存储到数据库中
4. 管理员可以查询和分析操作日志
5. 系统定期清理过期日志，保持数据库性能

## 6. 系统测试

本章在已完成开发与联调的基础上，从测试环境、测试策略出发，对新产品试制试验任务管理系统开展功能测试、性能与并发测试，并对测试结果进行归纳分析，以验证系统是否满足第 3 章提出的功能性与非功能性需求。

### 6.1 测试环境与测试策略

**（1）测试环境**

测试环境在逻辑上与生产环境保持一致，采用前后端分离部署：后端为基于 Spring Boot 的应用服务，数据库为 MySQL，缓存为 Redis；Web 前端通过浏览器访问；移动端通过 uni-app 编译产物在真机或模拟器上验证。硬件方面，服务端与客户端均使用满足 JDK、Node 及数据库运行要求的常规配置即可支撑本系统测试；网络环境为局域网或本地回环，以保证接口调用稳定、便于抓包与日志分析。

软件与工具版本与第 2 章技术栈说明一致，例如 Spring Boot 2.5.15、Vue 2.6.12、MyBatis、JWT 等，避免因版本差异导致“环境型”缺陷干扰对业务逻辑的判断。

**（2）测试策略**

测试策略遵循“自底向上、先主后次、风险优先”的原则：

- **单元与接口层**：对关键服务方法、REST 接口进行针对性验证，保证数据校验、状态机跳转、版本号更新等逻辑正确。
- **系统与验收层**：以业务场景为主线，采用黑盒测试方法，覆盖流转卡全生命周期、零件分流、程序变更、移动端扫码、双模打印等核心路径；对权限、日志、超期预警等横切能力进行抽样与回归。
- **非功能层**：针对第 3.3 节提出的性能与并发、数据一致性要求，设计性能与并发专项用例，重点验证乐观锁在并发写场景下的表现及接口响应时间是否落在预期区间。

测试过程中以《需求规格说明》与第 3 章需求分析为预期结果依据，缺陷记录包含复现步骤、严重级别与处理状态，便于回归与版本闭环。

### 6.2 功能测试

功能测试围绕试验任务与质量信息流转卡核心业务展开，验证各模块协同是否形成完整业务闭环。

#### 6.2.1 流转卡全流程闭环测试

本项测试验证从任务启动到流转卡归档的端到端行为，与第 3.1.2 节质量信息流转卡业务模型及第 4.2.3 节工序状态机设计相对应。典型步骤包括：在项目管理与任务模块中创建试验任务并生成流转卡；按工序顺序推进状态；在各节点录入质量信息与检验结果；执行必要的审批或确认操作；最终确认流转卡完成并支持查询与追溯。

测试要点包括：流转卡与任务数据的一致性；工序状态只能按规则前进或按程序变更规则回退（与第 4.4.3 节一致）；非法跳转应被拦截并提示；关键操作留有审计依据（与操作日志设计一致）。经测试，主路径可完整跑通，数据在库内与界面展示一致，满足“创建—采集—流转/审批—归档与追溯”的闭环要求。

例如，控制器在修改流转卡前若发现任务已结束则直接返回错误，从接口层阻断非法编辑（摘自 `TrialTaskProdController.java`）：

```java
@PutMapping
public AjaxResult edit(@RequestBody TrialTaskProd trialTaskProd) {
    TrialTaskProd oldNode = trialTaskProdService.selectTrialTaskProdByTaskId(trialTaskProd.getTaskId());
    if (TaskStatus.FINISHED.getCode().equals(oldNode.getStatus())) {
        return AjaxResult.error("任务已结束，无法修改");
    }
    return toAjax(trialTaskProdService.updateTrialTaskProd(trialTaskProd));
}
```

#### 6.2.2 零件分流场景专项测试

零件分流对应第 3.1.3 节与第 4.2.2、5.2.2 节所述“一码多卡”及复制、状态重置策略。专项测试用例覆盖：在符合业务条件时发起分流；分流后生成独立子任务/流转卡并与主任务保持可追溯关联；子卡工序初始化与状态重置是否符合设计；主任务与子任务在列表、详情、汇总视图中的展示是否正确。

测试中还关注异常与边界情况，例如重复分流、在不允许状态下发起分流等，系统应给出明确提示且不破坏已有数据完整性。结果表明，分流场景下数据结构与关联关系正确，与需求中“分流任务独立执行与监控”的描述相符。

#### 6.2.3 移动端扫码与填报测试

移动端测试在 uni-app 客户端进行，对应第 3.2.3 节与第 5.3 节实现。测试内容包括：扫描任务二维码后能正确解析并进入对应任务；自动定位到当前待填工序或待办环节（与扫码自动定位算法一致）；表单字段与 Web 端业务规则一致；未授权用户无法越权填报；已锁定或已完结工序不可编辑（状态控制）。

同时在弱网或重复提交场景下观察提示与数据是否重复入库。测试表明，扫码入口可用、定位逻辑正确，移动端与后端接口协同正常，满足现场填报与移动化管理需求。

#### 6.2.4 双模打印功能测试

双模打印包括基于 Thymeleaf 的网页即时预览与基于 JasperReports 的 PDF 归档下载（第 2.6 节、第 4.4.4 节与第 5.5 节）。测试验证：同一业务数据在两种模式下内容一致、版式符合模板设计；PDF 生成成功且可下载保存；网页预览在主流浏览器下打印对话框可用、分页与样式无明显错乱；权限不足时无法导出或打印敏感报表。

测试结果表明，双模策略能够兼顾“快速预览打印”与“正式归档”两类场景，与需求分析一致。

### 6.3 性能与并发测试

#### 6.3.1 乐观锁并发冲突场景测试

系统对关键实体采用版本号字段配合 MyBatis 拦截器实现乐观锁（第 2.7.2 节、第 4.4.1 节与第 5.4.1 节）。本项测试模拟多用户或同一用户多终端在短时间内对同一条记录发起更新请求：先发起的提交应成功，版本号递增；后发请求应检测到版本不一致并失败或进入业务约定的冲突处理分支（如提示刷新后重试），数据库中不得出现静默覆盖。

该类测试用于验证高并发写场景下的数据一致性与第 3.3.2 节“冲突检测和处理”要求。实测中，并发更新同记录时未观察到错误覆盖，冲突可被正确识别，符合乐观锁设计预期。

#### 6.3.2 系统响应时间与稳定性测试

依据第 3.3.1 节性能指标，对典型操作进行抽样计时：列表与详情页加载、常用查询接口、报表生成与预览等。可采用浏览器开发者工具、服务端日志时间戳或压测工具记录响应时间分布；在持续一段时间内重复执行高频接口，观察错误率与资源占用是否稳定。

测试目的为确认系统在试制业务常态负载下响应可接受、无内存泄漏或连接池耗尽等稳定性问题。若个别接口在冷启动或大数据量查询时超出阈值，可通过索引优化、分页与缓存策略在后续迭代中改进，并在本章测试结果中如实记录。

### 6.4 测试结果分析

综合上述测试，可以得出如下结论：

（1）**功能完整性**：流转卡全流程、零件分流、程序变更相关路径、移动端扫码填报及双模打印等核心功能均可按需求工作，主流程闭环清晰，异常与权限场景有相应控制，满足第 3.2 节功能性需求的主体描述。

（2）**可靠性与一致性**：乐观锁并发场景下未出现数据静默覆盖，说明第 4.4.1 节所实现的并发控制机制有效；为后续在生产环境配置连接池大小、线程数与数据库隔离级别提供了实证依据。

（3）**性能与体验**：在测试环境硬件与网络条件下，典型页面与接口响应基本满足第 3.3.1 节提出的量级预期；若与指标存在偏差，应区分是环境资源限制还是算法与 SQL 问题并列入优化清单。

（4）**局限与后续工作**：本章测试侧重于功能与主要非功能项，大规模长时间压测、安全渗透测试与多地容灾等可作为上线前或运维阶段的补充；随着用户量与数据量增长，可建立周期性回归与监控告警，与第 5.6 节日志审计、超期预警等机制共同保障系统可持续运行。

总体而言，当前版本已达到毕业设计阶段预定的测试目标，可为第 7 章总结与展望提供实证支撑。
